AUS1 – Príprava na obhajobu

Komplexný sprievodca semestralnou prácou: analýza sieťových tokov z botnet datasetu.

Čo program robí: Načíta sieťové toky zo CSV súboru a umožňuje ich filtrovať (Úr.1), prechádzať cez IP hierarchiu (Úr.2), rýchlo vyhľadávať podľa zdrojovej IP (Úr.3) a triediť (Úr.4). Všetky dátové štruktúry pochádzajú z knižnice libds.

struct IP

Reprezentuje IPv4 adresu. Nachádza sa v súbore data_models.hpp.

struct IP {
    std::array<uint8_t, 4> octets;  // 4 bajty: napr. 147.32.10.5
};

Prečo uint8_t a nie char?

Zadanie odporúča char octets[4]. My používame std::array<uint8_t, 4>.

Dôvod: char je na platforme znamienkový (signed). IP oktety môžu byť 0–255. Hodnoty väčšie ako 127 (napr. 192) by sa pri výpise zobrazili ako záporné číslo (192 = -64 v signed char). uint8_t je vždy neznamienkový bajt 0–255.

std::array je explicitne povolené zadaním ("okrem std::vector alebo std::array").

isInSubnet – kontrola podsiete

bool isInSubnet(const IP& net, int prefix) const {
    // prefix=16 → maska 255.255.0.0
    // Pre každý oktet vypočíta koľko bitov masky naň pripadá
    // a porovná zamaskované hodnoty.
    for (int i = 0; i < 4; ++i) {
        int bits = max(0, min(8, prefix - (i * 8)));
        if (bits == 0) break;
        uint8_t mask = 0xFF << (8 - bits);
        if ((octets[i] & mask) != (net.octets[i] & mask)) return false;
    }
    return true;
}
Príklad: IP 147.32.10.5, sieť 147.32.0.0, prefix=16
Oktet 0: bits=8 → mask=0xFF → 147&0xFF == 147&0xFF ✓
Oktet 1: bits=8 → mask=0xFF → 32&0xFF == 32&0xFF ✓
Oktet 2: bits=0 → break → true ✓

Načítanie dát – loadData()

Funkcia loadData v main.cpp načíta CSV a naplní tri dátové štruktúry naraz jediným prechodom cez súbor.

Fáza 1 – Čítanie hlavičky a colIdx lambda

Prvý riadok CSV je hlavička. Uložíme ho do SinglyLinkedSequence<std::string> colNames a vytvoríme lambdu colIdx, ktorá nájde index stĺpca podľa mena.

SinglyLinkedSequence<std::string> colNames;
// Hlavičku "DST_IP,SRC_IP,DST_PORT,..." rozdelíme podľa ',' a uložíme
while (std::getline(ssHeader, col, ','))
    colNames.emplaceLast().data = clean(col);

// Lambda prechádza colNames a hľadá index stĺpca podľa mena:
auto colIdx = [&](const std::string& name) -> int {
    int i = 0;
    auto* b = colNames.accessFirst();
    while (b) {
        if (b->data == name) return i;
        b = colNames.accessNext(*b);
        i++;
    }
    return -1;  // stĺpec nenájdený
};

int iSrc     = colIdx("SRC_IP");    // napr. 0
int iDst     = colIdx("DST_IP");    // napr. 1
int iBytes   = colIdx("BYTES");     // napr. 5
// ... a tak ďalej pre každý stĺpec
Prečo? Poradie stĺpcov v CSV nie je zaručené. Namiesto pevných indexov nájdeme každý stĺpec podľa názvu raz, uložíme index, a ďalej používame číslo. colNames sa po načítaní indexov ďalej nepoužíva.

data.emplaceLast() — alokuje nový blok v SinglyLinkedSequence, vráti referenciu na blok. Adresa tohto bloku sa nikdy nezmení.

Vyplníme block.data (FlowRecord) — parsujeme všetky stĺpce pomocou vopred vypočítaných indexov.

hierarchy.insert(block.data) — hierarchia dostane referenciu na FlowRecord, ale interne uloží len &rec (pointer na ten blok).

flowTable.insertFlow(&block.data) — tabuľka dostane priamo pointer. Obidve štruktúry ukazujú na ten istý FlowRecord v pamäti.

Ak catch: data.removeLast() — chybný riadok vyhodíme. Keďže hierarchia ani tabuľka ešte nedostali pointer (výnimka nastala skôr), nie je čo upratovať.

Prečo auto& block (referencia) a nie auto block (kópia)?
emplaceLast() vráti referenciu na blok, ktorý existuje priamo v zozname. Ak by sme urobili auto block = data.emplaceLast() (kópia), mali by sme lokálnu premennú – a &block.data by bol pointer na zásobník, nie na zoznam. Hierarchia a tabuľka by ukazovali na neplatnú pamäť po skončení iterácie smyčky.
Dôležité: Záznamy FlowRecord existujú len v data (SinglyLinkedSequence). Hierarchia aj tabuľka uchovávajú len const FlowRecord* — žiadna duplicita dát.
main() ├── SinglyLinkedSequence<FlowRecord> data ← VLASTNÍK dát ├── IpHierarchy hierarchy ← len const FlowRecord* └── FlowSortedTable flowTable ← len const FlowRecord*

SinglyLinkedSequence – jednosmerne viazaný zoznam

Pochádza z libds/amt/explicit_sequence.hpp. Cvičenie 3.

Ako funguje interne

Každý prvok je blok (SinglyLinkedSequenceBlock) s dvoma časťami:

Block { data: FlowRecord, next: → Block } ↓ Block { data: FlowRecord, next: → Block } ↓ Block { data: FlowRecord, next: → nullptr } ← tail

Zoznam drží referencie na prvý a posledný blok. Vkladanie na koniec (emplaceLast()) je preto O(1) – netreba prechádzať celý zoznam.

SortedSequenceTable – zoradená sekvenčná tabuľka

Pochádza z libds/adt/table.hpp. Cvičenie 11.

Ako funguje interne

Vnútorne drží ImplicitSequence párov (kľúč, hodnota) usporiadaných podľa kľúča. Vyhľadávanie prebieha binárnym vyhľadávaním.

Index: [0] [1] [2] [3] ("109.74.55.27") ("147.32.84.165") ("184.232.1.1") ("38.0.0.1") ↓ ↓ ↓ ↓ FlowList* FlowList* FlowList* FlowList* (zoradené podľa kľúča — reťazca IP adresy)

Operácia nájdi – O(log n)

Binárne vyhľadávanie: porovnáva kľúč s prvkom v strede, zužuje rozsah na polovicu.

// Pseudokód binárneho vyhľadávania:
left = 0, right = n - 1
while (left <= right):
    mid = (left + right) / 2
    if key == table[mid].key: return table[mid]
    if key <  table[mid].key: right = mid - 1
    else:                     left  = mid + 1
Maximálny počet porovnaní = ⌈log₂(n+1)⌉
Pre n = 19 710 IP adries → max 15 porovnaní (namiesto 19 710 pri lineárnom)

Operácia vlož – O(n)

Vloženie nájde pozíciu (O(log n)), ale musí posunúť všetky prvky napravo o jedno miesto, aby zostala zachovaná zoradenosť. To je O(n).

Vkladanie O(n) je akceptovateľné, pretože sa vykonáva raz pri načítaní. Vyhľadávanie O(log n) sa opakuje pri každom dotaze – tam je úspora 99,9%.

Duplicitné kľúče

Viac tokov môže mať rovnakú zdrojovú IP. FlowSortedTable to rieši tak, že hodnota tabuľky je zoznam (FlowList*). Nový tok s existujúcim kľúčom sa pridá do existujúceho zoznamu.

QuickSort

Pochádza z libds/adt/sorts.hpp. Cvičenie 12. Triedi ImplicitSequence in-place.

Algoritmus krok po kroku

  1. Pivot – vyberie sa prvok v strede rozsahu: is.access(min + (max-min)/2)->data
  2. Partition – ľavý index ide doprava kým nenájde prvok ≥ pivot, pravý ide doľava kým nenájde prvok ≤ pivot. Prvky sa vymenia (std::swap).
  3. Rekurzia – zavolá sa rekurzívne na ľavú a pravú časť.
void quick(is, compare, min, max) {
    T pivot = is.access(min + (max - min) / 2)->data;
    left = min; right = max;
    do {
        while (compare(is[left],  pivot)) ++left;   // hľadaj prvok ≥ pivot
        while (compare(pivot, is[right])) --right;  // hľadaj prvok ≤ pivot
        if (left <= right) { swap(is[left], is[right]); ++left; --right; }
    } while (left <= right);
    if (min   < right) quick(is, compare, min,  right);
    if (left  < max)   quick(is, compare, left, max);
}

Zložitosť

PrípadZložitosťKedy nastane

Pivot v strede (nie na kraji) znižuje pravdepodobnosť najhoršieho prípadu na zoradených dátach.

Lambda funkcie v C++

Lambdy sú anonymné funkcie definované priamo na mieste volania. V projekte ich používame všade — ako predikáty, procesory aj komparátory.

Základná syntax

[capture](parametre) -> návratový_typ { telo }
          ↑              ↑                    ↑              ↑
     čo zachytíme   vstupné arg.        voliteľné        kód funkcie
Najjednoduchší príklad:
auto f = [](int x) { return x * 2; };
f(5); // → 10

Lambda ako predikát (vracia bool):
auto pred = [](const FlowRecord& r) { return r.bytes > 1000; };
pred(rec); // → true alebo false

Capture list – čo zachytíme z okolia

Lambdy môžu používať premenné z vonkajšieho kontextu. To, čo chceme použiť, musíme zachytiť (capture) v hranatých zátvorkách.

ZápisČo znamenáPríklad z projektu
[]Nič nezachytíme — lambda závisí len od parametrov[](const FlowRecord& r) { return true; }
[x]Zachyť x hodnotou — kópia, zmena x vo vnútri neovplyvní vonkajšiu x[minB, maxB](const FlowRecord& r) { return r.bytes >= minB; }
[&x]Zachyť x referenciou — zmena vo vnútri zmení vonkajšiu x[&count](const FlowRecord& r) { count++; }
[&]Zachyť všetko referenciou[&](const std::string& name) { ... colNames ... } (colIdx)
[=]Zachyť všetko hodnotouV projekte nepoužívame

Kedy hodnotou a kedy referenciou?

Hodnotou [x] — keď chceš prečítať vonkajšiu hodnotu, ale nechceš ju meniť. Napr. minBajtov, maxBajtov, siet, prefix. Lambda si uloží kópiu v momente vytvorenia.

Referenciou [&x] — keď chceš meniť vonkajšiu premennú. Napr. count — procesor ho zvyšuje pri každom výpise. Bez & by sa zvyšovala len lokálna kópia a vonkajší count by zostal 0.

Ako lambda putuje do processData

// Definícia processData — Pred a Proc sú "nejaké typy" (šablóna)
template<typename Iter, typename Pred, typename Proc>
void processData(Iter begin, Iter end, Pred predicate, Proc processor);

// Volanie — kompilátor odvodí typy automaticky:
analyzer.processData(
    data.begin(), data.end(),

Každá lambda má unikátny typ, ktorý generuje kompilátor. Preto musí byť processData šablóna — nedá sa napísať ako normálna funkcia s konkrétnym typom parametra. std::function<bool(const FlowRecord&)> by fungovalo tiež, ale je pomalšie (dynamická dispatch) a menej elegantné.

IP Hierarchia – 4-úrovňová trie

Súbor ip_hierarchy.hpp. Najkomplexnejšia časť projektu.

Koncept

Každá zdrojová IP adresa tvorí cestu od koreňa po list v strome. IP 147.32.84.165:

koreň (level 0) → 147 (level 1) → 32 (level 2) → 84 (level 3) → 165 (level 4 = list) └── [FlowRecord*, FlowRecord*, ...]

IpHierarchyNode – štruktúra a konštruktor

struct IpHierarchyNode {
    int     level      = 0;   // 0=koreň, 1-3=vnútorný, 4=list
    uint8_t octetValue = 0;   // hodnota tohto oktetu (0-255)
    IpHierarchyNode* parent = nullptr;

    ImplicitSequence<IpHierarchyNode*>           children; // 256 slotov
    SinglyLinkedSequence<const FlowRecord*> records;  // len v listoch

    IpHierarchyNode(int lvl = 0, uint8_t val = 0, IpHierarchyNode* par = nullptr)
        : level(lvl), octetValue(val), parent(par)
    {
        if (lvl < 4) {                       // listy (level 4) nemajú deti
            for (int i = 0; i < 256; i++)
                children.emplaceLast().data = nullptr;  // 256 nullptr slotov
        }
    }
};
Konštruktor naplní children 256-krát nullptr. Každý slot zodpovedá jednej možnej hodnote oktetu (0–255). Slot je obsadený (!= nullptr) až keď sa pre ten oktet naozaj vytvorí syn. Toto nám dáva O(1) prístup: children.access(147) = priamo slot č.147.
Listy (level 4) children nemajú. Podmienka if (lvl < 4) zabráni zbytočnej alokácii 256 slotov v každom liste – list ukladá záznamy do records, nie deti.

getOrCreateChild – krok za krokom

IpHierarchyNode* getOrCreateChild(uint8_t octet) {
    auto* b = children.access(octet);   // O(1) – slot č. octet
    // b je pointer na blok ImplicitSequence, b->data je IpHierarchyNode*
    if (b->data) return b->data;         // syn existuje → vráť ho
    b->data = new IpHierarchyNode(level + 1, octet, this);
    return b->data;                      // nový uzol
}

children.access(octet) — O(1) prístup do poľa, vráti pointer na blok na pozícii octet.

Ak b->data != nullptr: uzol pre tento oktet už existuje, vrátime ho.

Ak b->data == nullptr: vytvoríme new IpHierarchyNode(level+1, octet, this) a uložíme pointer do slotu. Nový uzol má level o 1 vyšší, pamätá si svoju hodnotu oktetu a ukazovateľ nahor na nás (this).

IpHierarchy::insert – vloženie záznamu krok za krokom

void insert(const FlowRecord& rec) {
    IpHierarchyNode* cur = root_;       // začni v koreni (level 0)
    for (int i = 0; i < 4; ++i)
        cur = cur->getOrCreateChild(rec.srcIp.octets[i]);
    // cur je teraz list (level 4)
    cur->records.emplaceLast().data = &rec;  // pointer, nie kópia!
}

class IpHierarchyIterator {
    IpHierarchyNode* current_;
public:
    bool moveUp() {
        if (current_->isRoot()) return false;
        current_ = current_->parent;
        return true;
    }
    bool moveToChild(uint8_t octet) {
        auto* b = current_->children.access(octet);
        if (!b || !b->data) return false;  // syn neexistuje
        current_ = b->data;
        return true;

SubtreeDfsIterator – DFS prechod podstromom

Prechádza celý podstrom do hĺbky (DFS, preorder). Interná trieda používaná cez IpRecordIterator.

SubtreeDfsIterator& operator++() {
    // 1. Krok dolu (do prvého syna)
    if (!current_->isLeaf()) {
        for (i = 0...255)
            if (children[i]) { current_ = children[i]; return; }
    }
    // 2. Krok nabok (ďalší súrodenec) alebo nahor
    while (current_ != subtreeRoot_) {
        par = current_->parent;
        for (i = octetValue+1...255)
            if (par->children[i]) { current_ = par->children[i]; return; }
        current_ = par;  // žiadny súrodenec, krok nahor
    }
    current_ = nullptr;  // koniec
}

insertFlow – krok za krokom

void insertFlow(const FlowRecord* rec) {
    const std::string key = rec->srcIp.toString();  // napr. "147.32.84.165"

    FlowList** listPtr = nullptr;  // pointer na pointer

    if (this->tryFind(key, listPtr)) {
        // kľúč existuje → *listPtr je FlowList* → pridaj do existujúceho zoznamu
        (*listPtr)->emplaceLast().data = rec;
    } else {
        // nový kľúč → vytvor zoznam, vlož prvý tok, vlož do tabuľky
        FlowList* newList = new FlowList();
        newList->emplaceLast().data = rec;
        this->insert(key, newList);   // do SortedSequenceTable
    }
}

Vytvor kľúč: key = "147.32.84.165" (reťazcová reprezentácia srcIp).

tryFind(key, listPtr) — hľadá kľúč binárnym vyhľadávaním. Ak nájde, nastaví listPtr tak, aby ukazoval na hodnotu v tabuľke (FlowList* v tabuľkovom slote). Vráti true.

Ak true: *listPtr = dereferencia = samotný FlowList*. Pridáme ďalší tok do existujúceho zoznamu.

Čo je FlowList** listPtr a prečo pointer na pointer?

SinglyLinkedSequence – operácie krok po kroku

emplaceLast() – vloženie na koniec

Alokuje sa nový blok v pamäti: new Block { data, next=nullptr }

Aktuálny tail (posledný blok) nastaví svoj next ukazovateľ na nový blok.

tail zoznamu sa posunie na nový blok. Veľkosť size_ sa zvýši o 1.

Pred
[A] → [B] → [C] → nullptr    head=A, tail=C

Po emplaceLast() → nový blok D
[A] → [B] → [C] → [D] → nullptr    head=A, tail=D
Prečo O(1)? Zoznam drží referenciu na tail – priamy prístup, žiadne prechádzanie.

accessFirst() / accessNext() – iterácia

accessFirst() vráti ukazovateľ na head blok – O(1).

accessNext(block) vráti block.next – O(1), len nasleduje pointer.

Keď next == nullptr, iterácia skončila.

[A] → [B] → [C] → nullptr
accessFirst() = A
accessNext(A) = B
accessNext(B) = C
accessNext(C) = nullptr → koniec

removeLast() – odobratie posledného – O(n)!

Musíme nájsť predchodcu tail bloku – prejdeme celý zoznam od head až kým current.next == tail. Toto je O(n).

Predchodcovi nastavíme next = nullptr.

Starý tail sa uvoľní (delete). tail sa posunie na predchodcu.

Pred
[A] → [B] → [C] → nullptr    tail=C

Po removeLast()
[A] → [B] → nullptr    tail=B    C uvoľnený

Kontrola originality – Presun na najľavejší uzol v rovnakej úrovni

Čo môžu chcieť: „Naprogramuj metódu, ktorá z aktuálnej pozície presunie iterátor na najľavejší uzol v celom strome na rovnakej úrovni." Teda ak sme na úrovni 2, nájdi prvý existujúci uzol level=2 v celom strome (ideme vždy do ľavého potomka od koreňa).

Algoritmus

Zapamätaj si targetLevel = current_->level.

Vráť sa do koreňa: opakuj moveUp() kým isRoot().

Zostupuj targetLevel-krát: pri každom kroku prejdi i=0..255 a zavolaj moveToChild(i) pre prvý existujúci slot (najmenší index = najľavejší syn).

Ak na niektorej úrovni nie je žiadny syn (prázdny strom) — vráť false.

// Pridáme ako metódu do IpHierarchyIterator, alebo ako voľnú funkciu:
bool moveToLeftmostAtSameLevel(IpHierarchyIterator& it) {
    int targetLevel = it.current()->level;

    // Krok nahor až do koreňa
    while (!it.current()->isRoot())
        it.moveUp();

    // Zostupujeme targetLevel-krát, vždy najľavejší syn
    for (int l = 0; l < targetLevel; l++) {
        bool found = false;
        for (int i = 0; i < 256; i++) {
            if (it.moveToChild((uint8_t)i)) { found = true; break; }
        }
        if (!found) return false;
    }
    return true;
}

targetLevel = 2

i=1: moveToChild(1) → false
...

Sme na najľavejšom uzle level=2 v celom strome. return true
Čo povedať: „Algoritmus má dve fázy: najprv sa vrátime do koreňa pomocou existujúcej moveUp(). Potom targetLevel-krát zostupujeme, pričom vždy prechádzame children od indexu 0 a berieme prvý existujúci. Vďaka tomu, že children je ImplicitSequence s 256 slotmi indexovanými hodnotou oktetu, je každý moveToChild O(1). Celková zložitosť je O(256 × targetLevel) = O(1) — konštanta 256×4."

Kontrola originality – Presun na list úplne napravo

Čo môžu chcieť: „Z aktuálnej pozície v hierarchii (hocijaký uzol, hociaká úroveň) prejdi na najprávejší list v jeho podstrome." Teda z aktuálneho uzla vždy chod do posledného existujúceho syna, kým nedosiahneš list (level=4).

Algoritmus

Ak sme už v liste (isLeaf()) — sme hotovi, vráť true.

Opakuj krok 1–2 kým nie sme v liste.

Ak uzol nie je list, ale nemá žiadnych synov — vráť false (poškodený strom — v praxi nenastane).

bool moveToRightmostLeaf(IpHierarchyIterator& it) {

Operácie – ako fungujú

Čo sa stane v pamäti keď zavoláš emplaceLast() na SinglyLinkedSequence?
Alokuje sa nový blok (new Block) s dátami a next=nullptr. Aktuálny posledný blok (tail) nastaví svoj next ukazovateľ na nový blok. Interná referencia na tail sa posunie na nový blok. Veľkosť sa zvýši o 1. Celé je to O(1) pretože zoznam drží priamu referenciu na tail – netreba prechádzať zoznam.
Prečo je removeLast() na SinglyLinkedSequence O(n) a nie O(1)?
Jednosmerne viazaný zoznam nedrží referenciu na predchodcu tail bloku (len na tail samotný). Aby sme mohli odpojiť posledný blok, musíme nájsť predposledný – a to vyžaduje prejsť celý zoznam od head. Preto O(n). Dvojsmerne viazaný zoznam (DoublyLinkedSequence) by mal removeLast() v O(1) vďaka spätným ukazovateľom.
Čo sa stane v pamäti keď zavoláš emplace(i) na ImplicitSequence?
Najprv sa skontroluje kapacita (ak treba, realloc). Potom sa všetky prvky od indexu i do konca posunú o jedno miesto doprava – kopíruje sa od konca dozadu, aby sa neprepísali. Na uvoľnené miesto na indexe i sa zapíše nový prvok. Veľkosť sa zvýši o 1. Je to O(n) kvôli posunu prvkov.
Čo sa stane v pamäti keď zavoláš remove(i) na ImplicitSequence?
Prvok na indexe i sa vymaže (destruktor sa zavolá). Potom sa všetky prvky od i+1 do konca posunú o jedno miesto vľavo – kopíruje sa od začiatku doprava. Veľkosť sa zníži o 1. Je to O(n) kvôli posunu. Kapacita (alokovaná pamäť) sa nemení – pole sa nezmenšuje.
Ako funguje iterátor ImplicitSequence – čo je begin() a end()?
begin() vráti iterátor ukazujúci na prvý blok (accessFirst() = pointer na blok[0]). end() vráti iterátor ukazujúci za posledný blok (accessFirst() + size()). operator++ jednoducho posunie pointer o sizeof(MemoryBlock) bajtov – priama pointer aritmetika, veľmi rýchle, O(1).
Ako sa líši iterátor SinglyLinkedSequence od iterátora ImplicitSequence?
SinglyLinkedSequence – ExplicitSequenceIterator:
begin() = ExplicitSequenceIterator(m_first) (obalí pointer na prvý blok)
end() = ExplicitSequenceIterator(nullptr) (sentinel)
operator++: m_position = m_position->next – sleduje next pointer bloku

ImplicitSequence – ImplicitSequenceIterator:
begin() = pointer na blok[0]
end() = pointer za blok[size-1] (accessFirst() + size())
operator++: ++m_block – pointer aritmetika, posun o sizeof(MemoryBlock)

Kľúčový rozdiel: SinglyLinked sleduje reťaz next-pointerov (bloky nie sú v pamäti za sebou), ImplicitSequence sa pohybuje priamo v súvislom bloku pamäte.
Čo robí size() a isEmpty() a kde sú implementované?
Obe sú implementované v AbstractMemoryStructure – spoločnom predkovi všetkých libds štruktúr.

size() = deleguje na memory manager: memoryManager_->getAllocatedBlockCount(). Memory manager si drží počítadlo alokovaných blokov, takže size() je O(1).

isEmpty() = jednoducho return size() == 0, taktiež O(1).

V projekte sa isEmpty() používa napr. pri kontrole, či list v hierarchii nemá žiadne záznamy (preskočenie prázdnych listov v iterátore).
Vysvetli krok po kroku binárne vyhľadávanie v SortedSequenceTable.
Nastav left=0, right=n-1. Opakuj: mid=(left+right)/2. Porovnaj hľadaný kľúč s table[mid].key. Ak rovnaký → nájdené. Ak hľadaný < table[mid] → right=mid-1 (ľavá polovica). Ak hľadaný > table[mid] → left=mid+1 (pravá polovica). Keď left > right → nenájdené. Každý krok zúži rozsah na polovicu → O(log n).
Prečo je insert do SortedSequenceTable O(n) aj keď používa binárne vyhľadávanie?
Binárne vyhľadávanie nájde správnu pozíciu v O(log n). Ale potom musíme posunúť všetky prvky napravo od tejto pozície o jedno miesto, aby zostala tabuľka zoradená. Tento posun je O(n). Celková zložitosť je O(log n + n) = O(n). Vkladanie je pomalé, ale vykonáva sa len raz pri načítaní.
Ako QuickSort vyberá pivot a prečo práve stred?
Pivot je prvok na indexe min + (max-min)/2 – stred aktuálneho rozsahu. Prečo stred? Ak by sme brali pivot vždy z kraja (napr. index 0) a vstup by bol zoradený, vždy by sme dostali partition veľkosti 0 a n-1 → O(n²). Pivot v strede dáva lepšie výsledky na bežných dátach. Nie je to najlepší prístup (mediánový výber by bol lepší), ale je jednoduchý a prakticky funguje.
Čo je partition v QuickSort a ako funguje?
Partition rozdelí pole na dve časti: prvky ≤ pivot a prvky ≥ pivot. Robí to dvoma indexmi – left ide zľava doprava a hľadá prvok ≥ pivot, right ide zprava doľava a hľadá prvok ≤ pivot. Keď obaja nájdu svoje prvky a left ≤ right, vymenia ich (swap). Opakuje sa kým left > right. Po partition: ľavá časť (indexy 0..right) má prvky ≤ pivot, pravá (left..n-1) má prvky ≥ pivot. Rekurzívne sa utriedia obe časti.

Algoritmus a štruktúry

Čo je DataAnalyzer::processData a prečo je generický?
Je to šablónová metóda template<typename Iter, typename Pred, typename Proc>. Generická je preto, lebo neobsahuje žiadnu logiku viazanú na konkrétny typ dát. Funguje s akýmkoľvek iterátorom (SinglyLinkedSequence, ImplicitSequence, IpRecordIterator), akýmkoľvek predikátom a akýmkoľvek procesorom. Predikát a procesor sú odovzdané ako lambda funkcie v mieste volania.
Prečo ste použili SinglyLinkedSequence a nie ImplicitSequence pre hlavné úložisko?
SinglyLinkedSequence má O(1) vkladanie na koniec bez prealokácie – pri 160 000 záznamoch je to dôležité. Navyše, bloky existujú na stabilných adresách v pamäti, takže ukazovatele z hierarchie a tabuľky zostávajú platné po celý čas behu programu. ImplicitSequence môže pri raste realokovať pamäť a presunúť prvky, čím by ukazovatele stratili platnosť.
Prečo má IpHierarchyNode pole children s presne 256 slotmi?
Oktet IP adresy má hodnoty 0–255. Ak indexujeme pole priamo hodnotou oktetu, dostaneme O(1) prístup k synovi – stačí children.access(oktet). Alternatíva by bola zoradený zoznam alebo hash mapa synov, ale to by bolo pomalšie a komplikovanejšie. Nevýhoda je pamäť: aj keď existuje len 1 syn, alokujeme 256 slotov. V datasete je to akceptovateľné.
Aká je zložitosť načítania celej hierarchie a prečo?
O(n), kde n je počet záznamov. Každý záznam sa spracuje práve raz. Pre každý z 4 oktetov zdrojovej IP sa v O(1) nájde alebo vytvorí príslušný uzol (priamy prístup indexom do ImplicitSequence). Ukazovateľ na záznam sa vloží do listu v O(1). Súbor sa číta jediným prechodom.
Prečo je operácia nájdi O(log n) a vlož O(n) v SortedSequenceTable?
nájdi: Tabuľka je zoradená, takže môžeme použiť binárne vyhľadávanie – každý krok rozdelí rozsah na polovicu, takže max ⌈log₂(n+1)⌉ porovnaní.
vlož: Aj keď nájdeme pozíciu binárne (O(log n)), všetky prvky napravo od tejto pozície musíme posunúť o jedno miesto, aby zostala zachovaná zoradenosť. Posun je O(n).
Ako funguje QuickSort v libds – kde je pivot?
Pivot je vždy prvok v strede aktuálneho rozsahu: is.access(min + (max-min)/2)->data. Potom sa rozsah rozdelí: ľavý index ide doprava kým nenájde prvok ≥ pivot, pravý ide doľava kým nenájde prvok ≤ pivot. Nájdené prvky sa vymenia. Keď sa indexy stretnú, rekurzívne sa utriedia obe časti. Pivot v strede (nie na kraji) znižuje pravdepodobnosť O(n²) na zoradených dátach.
Prečo sa v Úrovni 4 záznamy kopírujú do ImplicitSequence a netriedí sa priamo hierarchia?
QuickSort potrebuje O(1) prístup indexom (is.access(i)). SinglyLinkedSequence nepodporuje O(1) prístup indexom – musel by prechádzať od začiatku, čo by triedenie znevýhodnilo na O(n²) len kvôli prístupu. ImplicitSequence je pole s O(1) prístupom, preto sa záznamy najprv skopírujú tam a potom utriedia.
Ako ste riešili duplicitné kľúče v tabuľke (viac tokov z rovnakej IP)?
Hodnota tabuľky nie je jeden FlowRecord, ale FlowList* – teda ukazovateľ na zoznam tokov (SinglyLinkedSequence<const FlowRecord*>). Pri insertFlow: ak kľúč existuje, pridáme do existujúceho zoznamu. Ak nie, vytvoríme nový zoznam a vložíme ho. Takto pod jedným kľúčom (IP adresou) môže byť ľubovoľný počet tokov.
Sú dáta v hierarchii a tabuľke kópie alebo originály?
Ani jedno. Obe štruktúry uchovávajú len const FlowRecord* – ukazovatele na originálne záznamy v SinglyLinkedSequence<FlowRecord> data. Žiadna duplicita dát. Záznamy existujú na jednom mieste a sú zmazané raz (keď sa destruuje data). Preto je data deklarovaná ako prvá v main() a destruuje sa ako posledná.
Prečo ste použili SortedSequenceTable a nie napr. HashTable?
SortedSequenceTable bola implementovaná na cvičení 11 a je súčasťou libds. Pre dataset ~19 710 unikátnych IP adries poskytuje O(log n) = max 15 porovnaní, čo je prakticky veľmi rýchle. HashTable by dala O(1) priemerné vyhľadávanie, ale SortedSequenceTable je jednoduchšia, bez kolízií, a pre tento dataset úplne dostačujúca. Za SortedSequenceTable je max 6 bodov, za HashTable/Treap max 10.
Sú v programe úniky pamäte?
Nie. Každý new má zodpovedajúci delete:
– new IpHierarchyNode() (koreň) → delete root_ v ~IpHierarchy()
– new IpHierarchyNode(...) (deti) → delete b->data v ~IpHierarchyNode() rekurzívne
– new FlowList() → delete item.data v ~FlowSortedTable()
Poradie deštrukcie v main() je bezpečné: tabuľka → hierarchia → dáta.
Prečo isInSubnet používa bitovú operáciu s maskou?
Maska pre prefix=16 je 255.255.0.0. Pre každý oktet vypočítame koľko bitov masky naň pripadá: pre oktet 0 to je min(8, 16-0)=8 bitov, pre oktet 1 min(8, 16-8)=8 bitov, pre oktet 2 min(8, 16-16)=0 bitov → break. Maska oktetu s k bitmi = 0xFF << (8-k). Potom porovnáme zamaskované hodnoty: ak (ip[i] & mask) == (net[i] & mask), oktety súhlasia v daných bitoch.
Čo je IpRecordIterator a ako prechádza záznamy?
Je to kombinovaný iterátor, ktorý zreťazuje záznamy zo všetkých listov podstromu. Interne drží SubtreeDfsIterator (na prechod uzlami) a recBlock_ (na prechod záznamami v aktuálnom liste). operator++ najprv skúsi posunúť recBlock_ v aktuálnom liste, ak je koniec, posunie DFS iterátor na ďalší uzol a preskočí prázdne listy. Výsledok: DataAnalyzer::processData ho používa rovnako ako akýkoľvek iný iterátor.
Prečo ste použili uint8_t namiesto char pre IP oktety?
char je na väčšine platforiem signed (znamienkový), teda reprezentuje hodnoty -128 až 127. IP oktety môžu byť 0–255. Hodnota napr. 192 by sa interpretovala ako záporné číslo (-64) a pri výpise by zobrazila záporné číslo. uint8_t je vždy neznamienkový bajt 0–255, čo je správne pre IP adresy. std::array je explicitne povolené zadaním.
Môže sa stať, že záznamy v hierarchii ukazujú na neplatnú pamäť?
Nie, ak sa program nezmení. Záznamy FlowRecord sú uložené v SinglyLinkedSequence<FlowRecord> data a nikdy sa nemažú počas behu. Hierarchia drží len const FlowRecord* – tieto ukazovatele zostávajú platné po celý čas, lebo bloky SinglyLinkedSequence sa nepresúvajú v pamäti (na rozdiel od ImplicitSequence, ktorá môže realokúvať). data je deklarovaná pred hierarchiou a destruuje sa po nej – takže záznamy existujú dlhšie ako ukazovatele.

Lambda funkcie

Čo je lambda funkcia a prečo ju používame?
Lambda je anonymná funkcia definovaná priamo na mieste volania. Syntax: [capture](parametre) { telo }. Používame ju preto, lebo predikáty a procesory sa líšia pri každom volaní processData — je pohodlnejšie napísať ich priamo ako argument než definovať desiatky pomenovaných funkcií. Algoritmus processData je šablóna, takže prijíma lambdy ako template parametre bez overhead-u.
Čo je capture list a aký je rozdiel medzi [x] a [&x]?
Capture list (hranatá zátvorka) určuje, ktoré vonkajšie premenné lambda "vidí".

[x] — hodnotou: lambda dostane kópiu x v momente vytvorenia. Zmena x vo vnútri lambdy neovplyvní vonkajšiu x. Používame pre minBajtov, maxBajtov, siet — len ich čítame.

[&x] — referenciou: lambda pracuje priamo s vonkajšou x. Zmena vo vnútri zmení aj vonkajšiu x. Používame pre count — procesor ho zvyšuje. Keby sme napísali [count], po skončení by count zostal 0.
Prečo processData berie lambdy cez šablónu (template) a nie cez std::function?
Každá lambda má unikátny typ generovaný kompilátorom — nedá sa vopred napísať konkrétny typ pre parameter. Šablóna template<typename Pred> nechá kompilátor odvodiť typ automaticky pri každom volaní.

std::function<bool(const FlowRecord&)> by fungovalo tiež, ale je pomalšie (typ sa erasuje, volanie ide cez virtual dispatch). Šablóna umožňuje inlining — kompilátor priamo vloží kód lambdy do smyčky, žiadne overhead volania funkcie.
Prečo Úroveň 4 používa std::function a ostatné úrovne nie?
V Úrovni 4 sa predikát vyberá v if-else vetve — rôzne vetvy priradzujú rôzne lambdy tej istej premennej predicate. auto nefunguje, lebo každá lambda má iný typ a auto musí mať fixný typ.

std::function<bool(const FlowRecord&)> predicate je "obal" schopný uložiť ľubovoľnú callable s rovnakým podpisom (bool, jeden argument FlowRecord). V ostatných úrovniach (1, 2) sa predikát odovzdáva priamo do processData bez medziskladovania — tam stačí auto alebo priama lambda.
Čo je closure a prečo lambda "vie" o count aj po skončení funkcie kde bola vytvorená?
Closure je lambda spolu so zachytenými hodnotami/referenciami. Ak zachytíme referenciou [&count], lambda drží pointer na count. Kým je count živý (na zásobníku runLevel1), lambda ho môže bezpečne meniť.

Nebezpečné by bolo, keby lambdu (s [&count]) uložíme a zavoláme PO skončení runLevel1 — vtedy count neexistuje a referencia je neplatná. V projekte to nenastáva, lebo processData sa dokončí skôr ako runLevel1 skončí.

Načítanie a hierarchia

Vysvetli krok po kroku čo sa stane pri načítaní jedného riadku CSV.
1. data.emplaceLast() — alokuje nový blok v SinglyLinkedSequence, vráti referenciu na blok.
2. Vyplní sa block.data (FlowRecord) — parsujeme všetky stĺpce pomocou vopred vypočítaných indexov (colIdx).
3. hierarchy.insert(block.data) — prejde 4 oktety srcIp, pre každý zavolá getOrCreateChild(oktet) → O(1) prístup do children. Na konci vloží &rec do records listu.
4. flowTable.insertFlow(&block.data) — vytvorí kľúč (srcIp.toString()), zavolá tryFind. Ak kľúč existuje, pridá pointer do existujúceho FlowList. Ak nie, vytvorí nový FlowList a zavolá insert.
5. Ak nastane výnimka (napr. chýbajúce pole): data.removeLast() — chybný záznam sa vyhodí.
Prečo sa v loadData používa auto& block (referencia) a nie auto block?
emplaceLast() vráti referenciu na blok, ktorý existuje priamo vo vnútri SinglyLinkedSequence. Ak by sme napísali auto block = data.emplaceLast(), mali by sme kópiu bloku na zásobníku. Potom &block.data by bol pointer na zásobník – ten je neplatný po skončení iterácie smyčky. Hierarchia a tabuľka by ukazovali na neplatnú pamäť. auto& zaručí, že pracujeme priamo s blokom v zozname a pointer &block.data je trvalý.
Čo robí colIdx lambda a prečo ju vôbec potrebujeme?
colIdx("SRC_IP") prechádza colNames (zoznam názvov stĺpcov z hlavičky) a vráti index stĺpca s daným názvom. Používame ju preto, lebo poradie stĺpcov v CSV nie je garantované – a ak by sa súbor zmenil, program by stále fungoval správne. Indexy sa vypočítajú raz pred hlavnou smyčkou a potom sa používajú priamo číselne (row.at(iSrc)), čím sa vyhneme opakovanému lineárnemu hľadaniu v každom riadku.
Čo presne robí getOrCreateChild a kedy vytvára nový uzol?
getOrCreateChild(octet): zavolá children.access(octet) – O(1) prístup do poľa 256 slotov. Slot na pozícii octet je buď nullptr (uzol neexistuje) alebo platný pointer (uzol existuje). Ak nullptr: vytvorí new IpHierarchyNode(level+1, octet, this) a uloží pointer do slotu. Ak nie nullptr: vráti existujúci uzol. Nový uzol si pamätá svoju hodnotu oktetu (octetValue) a ukazovateľ nahor (parent=this) – toto je potrebné pre DFS iterátor (hľadanie ďalšieho súrodenca) a pre výpis aktuálnej pozície v menu.
Prečo IpHierarchyNode zakazuje kopírovanie (= delete)?
Uzol drží parent pointer a children, ktoré uchovávajú pointre na dynamicky alokovaných synov. Keby sme povolili kopírovanie, kópia by zdieľala tie isté pointre – a delete pri destrukcii by mohol zavolať delete na ten istý uzol dvakrát (double-free). Zakázaním kopírovania (= delete) compiler zachytí akýkoľvek pokus o kopírovanie ako chybu kompilácie.

Tabuľka a tryFind

Vysvetli presne čo robí tryFind a prečo berie FlowList**.
tryFind(key, listPtr) hľadá kľúč binárnym vyhľadávaním. Ak ho nájde, nastaví listPtr tak, že *listPtr ukazuje priamo na hodnotu v tabuľkovom slote (teda na FlowList* uloženú v tabuľke) – nie na kópiu.

Prečo FlowList**? Keby sme dostali len FlowList* (kópiu), pridávanie do zoznamu by menilo lokálnu premennú, nie tabuľku. Cez pointer-na-pointer: (*listPtr)->emplaceLast() mení priamo objekt v tabuľke. Vráti true ak kľúč nájde, false ak nie (a listPtr zostane nezmenený).
Čo sa stane keď insertFlow zavolá insert(key, newList)?
this->insert(key, newList) volá metódu SortedSequenceTable::insert. Tá binárnym vyhľadávaním nájde správnu pozíciu kde má byť nový kľúč (aby zostala lexikografická zoradenosť). Potom posunie všetky prvky napravo o jedno miesto (O(n) posun v ImplicitSequence). Na uvoľnené miesto zapíše nový pár (key, newList). Toto je nákladné (O(n)), ale vykonáva sa len pri prvom výskyte každej IP adresy – každá IP sa opakuje mnohokrát, ale vkladá sa do tabuľky len raz.
Ako funguje vyhľadávanie v Úrovni 3 – čo presne volá tryFind?
std::string key = IP::fromString(ip).toString() — normalizuje vstup (napr. "109.74.55.027" → "109.74.55.27").
table.tryFind(key, listPtr) — binárne vyhľadávanie v zoradenom poli IP reťazcov, max 15 porovnaní.
Ak nájde: *listPtr je FlowList*. Voláme accessFirst() a accessNext() cez celý zoznam tokov.
Ak nenájde: vypíše správu "Ziadne zaznamy".
Celá operácia je O(log n) pre hľadanie + O(k) pre výpis, kde k je počet tokov pre túto IP.
